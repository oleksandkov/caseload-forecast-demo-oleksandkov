## Summary

Implemented `manipulation/4-train-IS.R` â€” the model estimation lane that fits two
forecasting models on Mint forge artifacts, evaluates them on a 24-month held-out
backtest window, and persists fitted objects plus a model registry CSV as the
primary hand-off contract for the Forecast lane.

---

## Architecture Decision: Full-Series Refit for Forecast Lane

**Decision**: Models stored in `.rds` are fitted on `ts_full` (all 246 months:
Apr 2005 â€“ Sep 2025), not `ts_train`.

**Rationale**: `5-forecast-IS.R`'s job is to project forward from the end of observed
data (Sep 2025 â†’ Oct 2025 onward). If `.rds` contained the `ts_train`-fitted model,
the "forecast" from Lane 5 would be issued from Sep 2023, not Sep 2025 â€” producing
a 2-year-stale horizon. The backtest evaluation (on `ts_train`) and the forecast
persistence (on `ts_full`) are therefore two separate steps in Train, each with its
own `set.seed(random_seed)` call.

**Implication for Lane 5**: `forecast(readRDS(rds_path), h = 24)` yields Oct 2025 â€“
Sep 2027. No re-fitting is required or permitted in Lane 5.

---

## Architecture Decision: Lightweight Model Registry CSV

**Decision**: `model_registry.csv` contains metadata and backtest metrics only â€” no
embedded forecast arrays, no prediction intervals, no backtest actuals.

**Rationale**: Embedding 24-row forecast arrays in the registry would couple Train
and Forecast more tightly than necessary, make the CSV unwieldy for human inspection,
and duplicate work that Lane 5 can produce freshly from the `.rds` file. The registry's
job is discovery and diagnostics â€” Lane 5's job is prediction.

**Contract**: Lane 5 reads the registry to discover which models exist, reads each
`.rds` via `rds_path`, and generates all forecast output independently.

## Backtest Protocol

1. Fit model on `ts_train` (222 months, Apr 2005 â€“ Sep 2023)
2. `forecast(fit, h = 24)` â†’ 24-month predictions covering `ts_test` period
3. `exp()` back-transform both predictions and actuals (undo Mint's `log()`, EDA-001)
4. Compute RMSE, MAE, MAPE on original caseload scale
5. Refit on `ts_full` â†’ save to `.rds`

**Backtest results** (Oct 2023 â€“ Sep 2025, original caseload scale):

| Tier | RMSE | MAE | MAPE |
|------|------|-----|------|
| Tier 1 â€” snaive | 10,300 | 9,545 | 16.4% |
| Tier 2 â€” ARIMA `(4,1,1)(1,0,0)[12]` | 8,639 | 7,831 | 13.3% |

ARIMA outperforms snaive by ~16% on RMSE. EDA-002's expected d=1 was confirmed by
`auto.arima()` on both train and full slices.

## Forecast Lane Hand-off: Step-by-Step Protocol for `5-forecast-IS.R`

### Step 1 â€” Validate forge lineage before loading any model

```r
forge_manifest  <- yaml::read_yaml(file.path(dir_forge, "forge_manifest.yml"))
model_registry  <- read.csv(file.path(dir_models, "model_registry.csv"))
current_hash    <- forge_manifest$forge_hash
registry_hash   <- model_registry$forge_hash[1]

if (current_hash != registry_hash) {
  stop(
    "forge_hash mismatch: models were trained on a different Mint run.\n",
    "  Registry hash : ", registry_hash, "\n",
    "  Current hash  : ", current_hash, "\n",
    "  Re-run 4-train-IS.R before forecasting."
  )
}
```

**Why this matters**: if `focal_date` changes in `config.yml`, Mint produces a new
manifest with a new hash, but `.rds` files from the old run still exist. This check
prevents silently using stale models.

### Step 2 â€” Load full series for ts reconstruction

```r
ds_full <- arrow::read_parquet(file.path(dir_forge, "ds_full.parquet"))
ts_full <- ts(ds_full$y,
              start     = c(year(min(ds_full$date)), month(min(ds_full$date))),
              frequency = 12)
# Note: ts_full is for reference/plotting only â€” models already contain ts_full internally
```

### Step 3 â€” Iterate over registry rows, load each model, generate forecast

```r
forecast_list <- list()

for (i in seq_len(nrow(model_registry))) {
  row   <- model_registry[i, ]
  model <- readRDS(row$rds_path)

  fc <- forecast(model, h = forge_manifest$mint_execution$forecast_horizon)
  # fc$mean        : point forecasts (log scale for ARIMA; original scale for snaive)
  # fc$lower[,"95%"] / fc$upper[,"95%"] : prediction intervals

  # Back-transform if log scale (check manifest, not hardcode)
  if (forge_manifest$transform_decisions$log_transform) {
    # snaive was fit on ts_full which is log-scale y â€” back-transform all outputs
    point     <- exp(as.numeric(fc$mean))
    lower_80  <- exp(as.numeric(fc$lower[, "80%"]))
    upper_80  <- exp(as.numeric(fc$upper[, "80%"]))
    lower_95  <- exp(as.numeric(fc$lower[, "95%"]))
    upper_95  <- exp(as.numeric(fc$upper[, "95%"]))
  } else {
    point    <- as.numeric(fc$mean)
    lower_80 <- as.numeric(fc$lower[, "80%"])
    upper_80 <- as.numeric(fc$upper[, "80%"])
    lower_95 <- as.numeric(fc$lower[, "95%"])
    upper_95 <- as.numeric(fc$upper[, "95%"])
  }

  # Build forecast dates: horizon after focal_date
  focal_date     <- as.Date(forge_manifest$mint_execution$focal_date)
  forecast_dates <- seq(focal_date %m+% months(1),
                        by = "month",
                        length.out = length(point))

  forecast_list[[row$model_id]] <- data.frame(
    model_id   = row$model_id,
    tier       = row$tier,
    tier_label = row$tier_label,
    date       = forecast_dates,
    point      = round(point, 0),
    lower_80   = round(lower_80, 0),
    upper_80   = round(upper_80, 0),
    lower_95   = round(lower_95, 0),
    upper_95   = round(upper_95, 0)
  )
}

# Combine all model forecasts into a single long-format data frame
ds_forecasts <- dplyr::bind_rows(forecast_list)
```

### Step 4 â€” Save forecast output and render report

```r
# Save combined forecasts as CSV
write.csv(ds_forecasts,
          file.path(dir_forecasts, "forecasts_IS.csv"),
          row.names = FALSE)

# Save as parquet for downstream report/dashboard consumption
arrow::write_parquet(ds_forecasts,
                     file.path(dir_forecasts, "forecasts_IS.parquet"))
```

## Important: snaive Forecast Mechanism

`snaive()` is fit with `h = forecast_horizon` at train time â€” the `forecast_horizon`
argument on `snaive()` sets the default look-ahead, NOT what `forecast()` uses.
When Lane 5 calls `forecast(fit_snaive_full, h = 24)`, it works correctly regardless
of what `h` was passed to `snaive()` at training time. The `h` on `snaive()` in
Train is redundant but harmless and documents intent.

---

## Important: Prediction Interval Back-Transformation

Both `snaive()` and `auto.arima()` are fitted on `y = log(caseload)`, so their
prediction intervals are on the log scale. Back-transforming via `exp()` gives
**median-unbiased** intervals on the original scale. This is industry-standard
for log-normal forecasting and is appropriate here.

`forecast::forecast()` returns `$lower` and `$upper` as matrices with columns
named `"80%"` and `"95%"`. Access them as:
```r
fc$lower[, "80%"]   # lower bound of 80% interval
fc$upper[, "95%"]   # upper bound of 95% interval
```
